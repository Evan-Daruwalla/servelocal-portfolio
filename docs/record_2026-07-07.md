## 2026-07-08 — M2: recurring events + waitlist

**WHAT:** The biggest domain gap vs v1. Opportunities can now recur weekly/monthly; students
subscribe to all dates or a single date, exclude/re-include individual dates, and a full one-time
opportunity waitlists new applicants and auto-promotes FIFO. Recurring hours auto-log one row per
past occurrence. Backend + frontend, 20 new tests (86 total, green); migration chain 0009–0010
applies up/down/up cleanly.

**WHY:** v2 previously handled one-time opportunities only. Orgs that run weekly/monthly programs
(the common case) couldn't be modeled, and a full event hard-rejected instead of waitlisting.

**HOW / design decisions (recorded per the PRD's "pick the simpler, record the choice"):**
- **Recurrence anchor derived from `start_time`**, not stored as separate day-of-week / day-of-month
  columns. This is exactly how v1 anchors (it steps from `startTime`), so it's the v1-faithful and
  DRY choice; only `recurrence` (one_time/weekly/monthly) + `series_end` are stored on Opportunity.
- **One application per (opportunity, user).** A student attends a subset of a recurring series via
  `excluded_dates` on an all-dates subscription, or picks exactly one date (`single_date`). v1's
  "several separate single-date applications to the same opp" is a scope reduction — the PRD success
  criterion (subscribe-all OR single-date, plus exclude) is fully met without it.
- **Occurrence math anchored to UTC** (`app/core/occurrences.py`), deliberately NOT porting v1's
  known UTC/local mixing bug. Monthly occurrences preserve the anchor day and clamp to a short
  month's last day computed from the original anchor each month (no drift): "monthly on the 31st"
  → Jan 31, Feb 28/29, Mar 31, Apr 30. 11 unit tests including leap-year Feb 29 and empty windows.
- **Capacity:** one-time uses the global `spots_remaining` counter + a FIFO waitlist
  (`app/services/enrollment.py::promote_from_waitlist`, called from withdraw and org-reject).
  Recurring uses per-occurrence-date capacity (`per_date_remaining`) surfaced by a new
  `GET /opportunities/{id}/date-spots`; no global counter, no waitlist (waitlisting is one-time only,
  per v1).
- **Hours:** the unique key moved from (opp, user) to (opp, user, occurrence_date) via a
  **batch-mode** migration (SQLite can't ALTER a constraint in place — batch recreates the table;
  verified down/up). Auto-log now expands past occurrences honoring exclusions, one pending row each,
  idempotent on reruns.

**Behavior change worth flagging:** a full one-time opportunity now returns 201 `waitlisted` instead
of 409. The existing `test_full_spots_rejects_new_applications` was updated to
`test_full_one_time_waitlists_new_applications` to assert the new, PRD-intended behavior.

**Verification:** 86 pytest green; frontend builds clean (15 routes); render-verified live — the
recurring detail page shows "Repeats weekly / Starts … (2h each), through 9/28/2026", the
subscribe-all vs single-date signup with live per-date availability from `/date-spots`, and the
create form's recurrence + conditional series-end controls. (Interactive click/submit through the
preview harness stayed flaky this session — verified writes directly against the API, which the 86
tests also cover.)

## 2026-07-08 — M3: hours parity (self-report, appeals, check-in codes)

**WHAT:** Completed the verified-hours loop. Three sources of Hours now: auto-logged, student
self-reported (`POST /hours`), and check-in-code redeemed (`POST /opportunities/{id}/checkin`).
A denied entry can be appealed once (`POST /hours/{id}/appeal` → status `appealed` → org
re-decides). Orgs generate per-date codes (`POST /opportunities/{id}/checkin-codes`). Backend +
frontend, 14 new tests (100 total, green); migrations 0011–0012 up/down/up clean.

**WHY:** v2 previously only auto-logged hours. v1 lets students self-report and appeal denials, and
run instant check-in via codes — the core "did you actually show up" verification loop.

**HOW / design decisions:**
- **Self-report is the simpler variant** (PRD-sanctioned): a student logs hours for an opportunity
  they have an application to; status `pending`; the org verifies via the existing route. v1's
  supervisor-email verification path is deferred — it needs the M4 email infra. Self-reported rows
  carry a **null `occurrence_date`** so they never collide with the per-occurrence unique index used
  by auto/check-in rows (nulls are distinct in a unique constraint).
- **Appeals:** a single-use flag (`appealed` bool + `appeal_note`); only `denied` rows are
  appealable, appealing flips status to `appealed` and notifies the org, and the org can then
  re-verify or re-deny. Guarded by tests (no double appeal, can't appeal verified).
- **Check-in codes** live in a JSON `checkin_codes` column on Opportunity ({date: code}), **never**
  exposed in any read schema (a leaked code = self-verified hours without attending — asserted by a
  test). Redemption requires an approved signup covering that occurrence date, is idempotent (409 on
  double check-in), and is guarded by an **in-memory per-user guess throttle** (10 fails / 15 min —
  single-process; a multi-process deploy would need a shared store, noted in `services/checkin.py`).
  Codes use an ambiguity-free alphabet (no O/0/I/1) for reading aloud at events; matched with
  `secrets.compare_digest`.

**Verification:** 100 pytest green; frontend builds clean; render-verified live end to end — seeded
a weekly opp, subscribed a student, the org generated a code and the student redeemed it via the API
(→ verified "check-in" hours for 2026-08-10), and the student hours page then showed "2 verified
hours" with a "Weekly Reading Buddies · 2h · Check-in · Verified" row plus the self-report and
check-in forms; the org detail page rendered the "Check-in codes" section with the occurrence-date
dropdown. (Form submit via the preview harness remained flaky; drove writes through the real API,
which the 100 tests also cover.)

## 2026-07-08 — M4: email infrastructure + password reset

**WHAT:** Shared transactional-email infra (`app/core/email.py`) and the first flow that uses it —
enumeration-safe password reset (`POST /auth/forgot`, `POST /auth/reset`). Backend + frontend, 7 new
tests (107 total, green); migration 0013 up/down/up clean; verified live end to end against a real
running server.

**WHY:** M4 precedes M5/M6 in the PRD because guardian consent and notifications both need email.
Password reset is the first consumer — it proves the infra and closes a real account-recovery gap.

**HOW / design decisions:**
- **Email is a logged no-op without `RESEND_API_KEY`** (dev/CI default) and POSTs to Resend's HTTP
  API when the key is set. httpx is imported **lazily** inside the real-send branch, so the stub path
  has no import-time dependency. **Fire-and-forget:** a send failure is logged and swallowed, never
  raised into the request — a user must not see a 500 because email is down. (A real key is
  BLOCKED-ON-EVAN; everything downstream works on the stub.)
- **Enumeration-safe forgot:** `POST /auth/forgot` always returns the same 200 body whether or not the
  email exists. Only a sha256 **hash** of a single-use `secrets.token_urlsafe(32)` token is stored on
  the user (+ 1h expiry); the raw token lives only in the emailed link. Per-identity in-memory throttle
  (3 req / 15 min) — same single-process caveat as the check-in throttle.
- **Session invalidation via `token_version`:** a new `int` column on User, embedded in the JWT as `tv`
  and compared in `get_current_user`. Reset bumps it, so **every JWT issued before the reset is
  rejected**. The token functions grew a `token_version` param (only two callers — login mints with the
  user's version, deps checks it). SQLite drops tz on the expiry column, so `reset` normalizes a naive
  stored value to UTC before comparing.
- **Frontend:** `/forgot` (email → generic confirmation) and `/reset?token=…` (new password; the
  `useSearchParams` read is wrapped in a `<Suspense>` boundary so the static build succeeds), plus a
  "Forgot your password?" link on the login page.

**Verification:** 107 pytest green (+7: single-use, expiry, enumeration-safe, old-JWT-rejected,
throttle-after-3). Migration 0013 up/down/up clean. Frontend lint + production build clean (both routes
prerendered static). **Live end-to-end** against a real uvicorn server on SQLite: register → forgot
(identical body for known vs unknown email) → read the reset token from the email stub's log → reset →
old password 401 / new password 200 → the pre-reset JWT now 401 → reusing the token 400. The browser
form itself wasn't driven (the preview harness's click/submit flakiness recorded in M2/M3); the pages
use the same component pattern as the browser-verified login page, and the live server proves the wiring.

## 2026-07-08 — M5.2 + M5.3: guardian-consent flow + gate (M5 complete — LAUNCH GATE)

**WHAT:** Finished the launch gate. Backend (M5.2): `app/api/routes/consent.py` —
`POST /consent/request` (student resends the guardian email, throttled 1/5min), public
`GET/POST /consent/{token}` (approve/decline, single-use, 410 on expiry), public

public `/consent/[token]` (approve/decline) and `/consent/manage/[token]` (revoke) pages that need

**WHY:** M5 is the hard launch gate — no public launch with minors before a guardian can approve
(and revoke). The gate is a single reusable dependency so M7 messaging inherits it for free.

**HOW / design decisions:**
- **The gate is `consent_blocks(user)` in `app/core/consent.py`**, duck-typed (core stays
  model-free) and recomputing age live every call: an 18+ student is never gated regardless of
  stored status, and a pending minor who ages into 18 is unblocked with no migration (unit-tested
  directly with `SimpleNamespace`, since it can't be reached via the API without a back-dated dob).
- **Tokens**: only sha256 hashes stored; the raw one-time approve token (72h TTL, `secrets.
  token_urlsafe(32)`) and a separate long-lived manage token live only in the emailed links.
  Approve clears the one-time hash (single use) and mints the manage token; the manage link ships in
  the approval confirmation email so the guardian can revoke later.
- **Public responses expose only first name + last initial** (v1 spec §4) — never email/dob/location.
- **Route order** in the consent module puts `/request` and `/manage/{token}` before the bare
  `/{token}` so Starlette doesn't capture the literals as a token. **Bug caught in verification:** the
  consent router was imported in `router.py` but never `include_router`'d — every consent call 404'd
  (and `test_context_unknown_token` passed for the wrong reason). Fixed by mounting it; all 11 tests
  then pass for the right reason.
- **First email is via `/consent/request`, not register** — register (M5.1) only records `pending`;
  the request endpoint (which is also the resend) generates the token and sends. The frontend banner
  drives it.

### ServeLocal v2 — Current State (2026-07-07)

Always-current snapshot of where the v2 rewrite stands. No history — for the chronological trail see
`record_2026-07-07.md`. When architecture or scope shifts significantly, create a new
`state_<date>.md` and mark this one superseded.

#### What it is
Production-stack rewrite of the v1 volunteer-opportunity platform. Students discover/track service,
log verified hours, earn awards; orgs post opportunities, manage applicants, verify attendance.
Students are free forever; orgs will have free/paid plans (M8). Driven by `PRD_ROADMAP.md`.

#### Architecture (as built)
- **Backend:** FastAPI + SQLAlchemy 2.0 (sync sessions) + Alembic + PostgreSQL (SQLite for tests).
  - `app/main.py` app factory; `app/core/` (config, security, awards); `app/db/` (session, base);
    `app/models/`, `app/schemas/`, `app/services/`; `app/api/router.py`, `app/api/deps.py`,
    `app/api/routes/`.
  - Auth: argon2 password hashing (`argon2-cffi`), HS256 JWTs (`PyJWT`) with `sub` + `exp`.
  - Models register in `app/models/__init__.py` (NOT `db/base.py` — circular import). Import order
    matters: application/hours/saved/review relationships reference Opportunity.
  - Config anchors `.env` to the backend dir via `Path(__file__)`, not cwd (a launcher-cwd bug fixed
    during the auth slice).
- **Frontend:** Next.js 15 (App Router) + React 19 + TypeScript + Tailwind 3 + shadcn/ui.
  - `lib/api.ts` (typed fetch wrapper), `lib/auth-context.tsx` (AuthProvider/useAuth, token in
    localStorage under exported `TOKEN_KEY`), `lib/types.ts`, `components/ui/`.
  - Pages: home, login, register, discover, opportunities/[id] (+ reviews/messages sections),
    opportunities/new, applications, applicants, hours (+ awards), verify-hours, leaderboard, saved,
    notifications.

#### Data model (migrations 0001–0013)
users (+ token_version, reset-token hash/expiry) · opportunities (+ recurrence/series_end, checkin_codes JSON) · applications
(+ subscription_type/single_date/excluded_dates, status incl. waitlisted) · hours (+ occurrence_date
keyed per occurrence, + source/note/appeal_note/appealed) · saved_opportunities · reviews ·
notifications · messages.

## 2026-07-09 — M6: notifications email delivery + opt-out (M6 complete)

**WHAT:** Upgraded the early notifications feature to the PRD's M6 spec. Every in-app notification
now also sends an email unless the recipient opted out, an opt-out toggle exists end to end, and
the list endpoint is paginated.
- Backend: `email_notifications` bool on `User` (default true, migration `0015_user_email_notifications`);
  `create_notification` (in `app/services/notifications.py`) now fetches the user and fires
  `send_email(to, subject=title, body=body or title)` when the flag is on; `PATCH /auth/me`
  (new `UserUpdate` schema) toggles the flag; `email_notifications` added to `UserRead`;
  `GET /notifications` gained `limit`/`offset`. 4 new tests (email fires by default, opt-out
  suppresses it while the in-app row still lands, pref defaults true + persists, pagination). 131
  backend tests green (was 127).
- Frontend: `User.email_notifications` type, `api.updateMe`, and an accessible email opt-out toggle
  (label + `htmlFor`/`id`, `accent-primary`, optimistic write with revert-on-failure) on
  `app/notifications/page.tsx`. `npm run lint` + `npm run build` clean.

**WHY:** The trigger events were already wired — applications approve/reject, waitlist promotion,
hours verify/deny/appeal, and consent approve/revoke all call `create_notification`. So the whole
M6 "wire email into every event" requirement collapsed to adding the email dispatch in that ONE
helper instead of touching eight call sites. Surgical and DRY, and it means no future
`create_notification` caller can forget to email.

**HOW / tradeoff:** Email is dispatched inside `create_notification`, before the caller's commit.
`send_email` never raises and is a logged no-op without `RESEND_API_KEY`, so the at-most-once edge
(email sent, then the outer transaction rolls back) is an accepted, deliberately-simple tradeoff
rather than an outbox/queue. Noted here so it's a known limitation, not a silent one — revisit if
transactional-email guarantees ever matter (M9/M11).

## 2026-07-09 — M9.1: per-IP rate limiting (middleware)

**WHAT:** `app/core/rate_limit.py` — a `RateLimitMiddleware` doing an in-memory sliding-60s-window
count per (client IP, bucket). Two buckets: `auth` (any `/auth/*`, default 30/min) and `write`
(other POST/PUT/PATCH/DELETE, default 120/min); reads are unlimited. Over-limit → 429 with a
`Retry-After` header. Limits + an on/off switch are in config (`RATE_LIMIT_ENABLED`,
`RATE_LIMIT_AUTH_PER_MIN`, `RATE_LIMIT_WRITE_PER_MIN`). 5 tests (auth burst, retry-after, reads
free, write burst, disabled bypass). 164 backend green (was 159).

**HOW / decisions:** added the middleware BEFORE CORS in `create_app` so CORS is the outer layer —
a 429 still carries CORS headers for the browser. The window store is a module-global
`{(ip,bucket): deque}`; `reset()` clears it and the `conftest` `client` fixture calls it per test,
so the shared state never leaks between tests (the reason the defaults can stay generous). Verified
the middleware doesn't break the billing webhook's raw-body read (`await request.body()`) — the full
suite is green.

**Known limitation (stated, not hidden):** SINGLE-PROCESS only — the window is per-process memory,
so a multi-process/multi-instance deploy needs a shared store (Redis). And behind a proxy
`request.client.host` is the proxy IP; prod should trust a validated `X-Forwarded-For`. Both flagged
for M11. (Consistent with the other in-memory throttles: password-reset, consent-resend, check-in.)
